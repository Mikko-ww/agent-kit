# self-evolve 详细说明

## 1. 设计目标

`self-evolve` v4 的目标不是“自动把零散学习条目攒起来”，而是提供一条稳定、可追溯、可审核的项目级知识流水线：

```text
session -> candidate -> rule -> skill sync
```

这条流水线解决两个问题：

1. Agent 如何把一次任务中的事实、修复和经验结构化落盘。
2. 项目如何把这些事实提炼成经过确认的正式规则，并稳定提供给所有 Agent 消费。

## 2. 为什么彻底重构

旧版模型以 `learning` 为中心，依赖 `capture/analyze/promote/evolve` 这组命令，存在几个根本问题：

- “Agent 自主发现需要学习的知识”只停留在文案提示，没有真正的检测器。
- 推广条件与数据模型耦合过深，正常录入路径很容易卡住。
- Skill 输出、数据模型和 CLI 语义都围绕旧概念展开，难以继续扩展。

因此 v4 直接放弃旧命令、旧布局和旧状态机，不保留兼容层。

## 3. 核心实体

### 3.1 SessionRecord

`session` 是唯一主输入，记录一次任务中的结构化事实。

关键字段：

- `id`
- `created_at`
- `source`
- `summary`
- `domain`
- `outcome`
- `observations[]`
- `decisions[]`
- `fixes[]`
- `lessons[]`
- `files[]`
- `tags[]`
- `processed`

### 3.2 KnowledgeCandidate

`candidate` 是检测器从一个或多个 session 中提炼出的待确认知识。

关键字段：

- `id`
- `status`
- `title`
- `statement`
- `rationale`
- `domain`
- `tags[]`
- `confidence`
- `fingerprint`
- `source_session_ids[]`
- `derived_from`

### 3.3 KnowledgeRule

`rule` 是项目已批准的正式规则，是 Skill 消费侧唯一可见的知识层。

关键字段：

- `id`
- `status`
- `title`
- `statement`
- `rationale`
- `domain`
- `tags[]`
- `source_session_ids[]`
- `source_candidate_ids[]`
- `revision_of`

### 3.4 KnowledgeIndex

`index` 负责维护追溯关系和快速查找键，例如：

- `fingerprint -> candidate ids`
- `fingerprint -> rule ids`
- `session -> candidate ids`
- `candidate -> rule`
- `active rule by fingerprint`

## 4. 候选检测器

v4 首版检测器只消费显式 session 输入，不扫描工作区，不接入 LLM。

### 4.1 候选生成规则

- 优先从 `lessons[]` 生成 candidate。
- 如果 `lessons[]` 为空，则按 `observation + fix` 配对生成 candidate。
- `fingerprint = normalize(domain + statement)`。

### 4.2 置信度

- `0.70`：来自显式 `lesson`
- `0.45`：来自 `observation + fix`
- `+0.10`：`outcome=success`
- `+0.15`：同 fingerprint 已在其他 session 出现
- `+0.05`：同 fingerprint 已存在 open candidate
- 总分上限 `1.0`

### 4.3 自动生效

默认关闭。

只有在同时满足下面条件时，candidate 才会在 `detect run` 阶段自动转成 rule：

- `auto_accept_enabled = true`
- `confidence >= auto_accept_min_confidence`
- 当前 fingerprint 没有 active rule

## 5. CLI 工作流

### 5.1 初始化

```bash
agent-kit self-evolve init
```

### 5.2 记录 session

```bash
agent-kit self-evolve session record \
  --summary "修复启动校验" \
  --domain debugging \
  --outcome success \
  --lesson "服务启动前必须先校验必填环境变量"
```

### 5.3 运行检测

```bash
agent-kit self-evolve detect run
```

### 5.4 审核 candidate

```bash
agent-kit self-evolve candidate list
agent-kit self-evolve candidate accept C-001
agent-kit self-evolve candidate reject C-002
```

### 5.5 直接录入 rule

```bash
agent-kit self-evolve rule add \
  --title "启动前校验环境变量" \
  --statement "在服务启动前校验所有必填环境变量。" \
  --rationale "避免进入部分启动成功、运行时再失败的状态。" \
  --domain debugging
```

### 5.6 同步 Skill

```bash
agent-kit self-evolve sync
```

## 6. 存储布局

```text
.agents/self-evolve/
├── config.jsonc
├── sessions/
├── candidates/
├── rules/
└── indexes/
```

Skill 输出位于：

```text
.agents/skills/self-evolve/
├── SKILL.md
├── catalog.json
├── scripts/find_rules.py
└── domains/
```

## 7. Skill 同步约束

- 只同步 `status=active` 的正式 rule。
- 不同步 candidate。
- 不同步 session。
- `catalog.json` 使用 v2 结构，包含 `statement`、`rationale`、`source_sessions`、`source_candidates` 等字段。

## 8. 旧格式处理

下面这些旧格式全部视为废弃：

- `.agents/self-evolve/learnings/`
- `.agents/self-evolve/rules.jsonc`
- 旧 `capture/analyze/promote/evolve/search` CLI

一旦发现旧布局，插件会直接报错并终止；不会尝试迁移，也不会读取旧数据。

## 9. 推荐验证

开发或改动本插件后，至少运行：

```bash
uv run pytest packages/self-evolve/tests -q
```

如果改动了 Skill 模板或检索脚本，还要额外验证：

```bash
python .agents/skills/self-evolve/scripts/find_rules.py --stats
python .agents/skills/self-evolve/scripts/find_rules.py --detail --keyword "..."
```
