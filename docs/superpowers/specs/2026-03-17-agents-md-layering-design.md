# AGENTS.md 分层设计

## 摘要

当前仓库只有根目录一个 `AGENTS.md`，随着 core、插件和插件数量增长，根文件会逐步变得臃肿，并且任何局部规则调整都会导致根文件频繁变化。

本设计将 `AGENTS.md` 拆成四层：

- 根目录 `AGENTS.md`
- `src/agent_kit/AGENTS.md`
- `packages/AGENTS.md`
- `packages/<plugin>/AGENTS.md`

核心原则是：

- 父级规则默认继承
- 子级只做补充和细化
- 子级不覆盖父级中的全局规则
- 根目录只保留全局规则、总体架构摘要和目录导航

## 目标

- 让每个 `AGENTS.md` 各司其职，避免单文件过大
- 降低局部规则调整对根目录文档的影响
- 让 agent 在进入具体目录时优先看到与当前任务强相关的约束
- 保持跨目录规则的一致性，避免局部说明和全局规范冲突

## 非目标

- 本次不引入更细颗粒度的 `tests/AGENTS.md` 或 `docs/AGENTS.md`
- 本次不改变代码结构，只调整 agent 说明文档结构
- 本次不允许子级 `AGENTS.md` 覆盖父级全局规则

## 分层结构

### 1. 根目录 `AGENTS.md`

职责：

- 定义全局规则
- 给出项目总体架构摘要
- 提供目录级导航
- 说明 `AGENTS.md` 的分层继承关系

建议保留内容：

- 语言规范
- Git 提交规范
- Markdown 与说明文档规范
- 项目总体架构摘要
- 顶层目录导航
- 指向 `src/agent_kit/AGENTS.md` 和 `packages/AGENTS.md` 的说明

不应继续放入的内容：

- 某个具体插件的业务细节
- 某个插件的命令行为细节
- 某个目录下的局部实现说明

### 2. `src/agent_kit/AGENTS.md`

职责：

- 描述 core 的职责边界
- 描述 CLI、插件管理器、注册表和路径布局
- 约束修改 core 时的注意事项

建议包含内容：

- 继承根目录规则的说明
- core 的命令模型
- `PluginManager` 的职责
- 注册表模型与双注册表文件同步要求
- 配置、数据、缓存路径规则
- 安装、更新、运行时校验要点
- core 相关测试入口

### 3. `packages/AGENTS.md`

职责：

- 描述所有插件共享的协议和命名约束
- 给新增插件提供统一 checklist

建议包含内容：

- 继承根目录规则的说明
- 插件分发名、目录名、模块名、`plugin_id` 的命名约定
- 默认不再使用 `agent-kit-` 前缀
- 插件必须提供 `agent-kit-plugin`
- 插件必须支持 `--plugin-metadata`
- 插件配置结构与 `config_version`
- 新增插件 checklist
- 插件级测试要求

### 4. `packages/<plugin>/AGENTS.md`

职责：

- 描述单个插件自己的业务规则和边界
- 只面向该插件目录下的开发工作

首个实例是：

- `packages/skills-link/AGENTS.md`

建议包含内容：

- 继承上级规则的说明
- 插件目标与边界
- 命令列表与行为摘要
- 配置文件结构
- 业务规则与禁止行为
- 修改该插件时需要重点验证的测试

## 继承规则

文档关系固定为：

- 根目录 `AGENTS.md` 是最高层规则
- 子级 `AGENTS.md` 默认继承父级规则
- 子级只允许补充和细化
- 子级不允许声明与父级冲突的规则

推荐写法：

- 每个子级 `AGENTS.md` 开头用一句话说明“继承父级规则，本文只补充本目录特有约束”
- 再摘录 2 到 4 条本目录最关键、最容易误用的规则

这样做的目的是：

- 保留分层导航能力
- 避免根规则被重复大段复制
- 又能防止 agent 只读局部文件时遗漏关键约束

## 内容边界

### 根目录保留

- 全局规范
- 总体架构摘要
- 导航

### 下沉到 `src/agent_kit`

- core 的实现边界
- 注册表与插件运行模型
- core 测试入口

### 下沉到 `packages`

- 插件协议
- 插件命名
- 插件新增约束

### 下沉到具体插件目录

- 单插件命令
- 单插件配置结构
- 单插件业务规则
- 单插件测试重点

## 维护规则

- 根目录文档中的路径引用统一使用相对路径
- 新增插件时，必须同时新增该插件根目录的 `AGENTS.md`
- 如果修改的是插件共享协议，应优先更新 `packages/AGENTS.md`
- 如果修改的是某个插件自己的业务边界，应优先更新对应插件目录下的 `AGENTS.md`
- 如果修改影响全局语言、提交规范或总体架构摘要，才需要改根目录 `AGENTS.md`

## 建议落地顺序

1. 将当前根目录 `AGENTS.md` 精简为“全局规则 + 总架构摘要 + 导航”
2. 新增 `src/agent_kit/AGENTS.md`
3. 新增 `packages/AGENTS.md`
4. 新增 `packages/skills-link/AGENTS.md`
5. 检查各层文档中的路径均为相对路径
6. 检查是否有重复说明仍滞留在根目录

## 验收标准

- 根目录 `AGENTS.md` 明显比当前版本更精简
- agent 在 `src/agent_kit/` 下工作时，可以从局部 `AGENTS.md` 读到 core 约束
- agent 在 `packages/` 或某个插件目录下工作时，可以从局部 `AGENTS.md` 读到插件相关约束
- 新增插件时不需要继续向根目录堆叠插件细节
- 所有 Markdown 路径引用都使用相对路径
