# 面向 Agent / 大语言模型辅助的第一方插件方向建议

## Issue 标题

建议优先规划一批面向 Agent / 大语言模型辅助场景的第一方插件

## 背景

我希望 `agent-kit` 后续优先投入的第一方插件，不是继续沿着通用开发环境工具去扩展，而是更明确地服务 **Agent 工程** 和 **大语言模型辅助开发**。

换句话说，插件方向应该更贴近下面这些问题：

- 如何让 Agent 更好地理解当前项目与工作区上下文
- 如何让 Agent 更稳定地复用技能、提示词、工具和工作流
- 如何让 Agent 的执行过程更容易调试、回放、评测和优化
- 如何让本地工具、MCP 服务和多 Agent 协作更容易接入

因此，这一轮插件方向不需要再以当前两个第一方插件为主要参考，而应当直接从 Agent 工具链本身出发来设计。

## 目标

- 把 `agent-kit` 往 **Agent 工具平台** 的方向推进，而不仅是通用 CLI 插件平台
- 优先补足 Agent 在本地使用时最缺的几类能力：上下文、工具接入、评测、记忆、回放、协作
- 让插件既能辅助现有 Agent，也能服务未来更多基于大语言模型的 Agent 工作流

## 调研结论

从当前 Agent 工具链的常见能力来看，更适合作为第一方插件重点投入的方向主要有以下几类：

- **Workspace Context**：项目结构理解、上下文打包、知识摘要
- **Agent Memory**：跨会话记忆、偏好记录、项目决策沉淀
- **Prompt / Skill Library**：提示词模板、技能包、可复用工作流
- **Tool Orchestration / MCP Integration**：工具注册、协议接入、服务健康检查
- **Evaluation**：效果评测、回归测试、Prompt / Tool 策略比较
- **Session Replay**：执行轨迹记录、回放、问题定位
- **Multi-Agent Workflow**：任务拆分、角色协作、执行编排

这些能力都比较适合做成插件，而不是直接塞进 core。原因是它们都有独立配置、独立生命周期，而且不同团队的工作流差异会非常大，更适合以插件方式演进。

## 候选插件方向

### 1. `agent-context`

用途：为 Agent 自动准备当前工作区的高价值上下文。

建议能力：

- 扫描当前仓库结构、关键目录、README、配置文件
- 自动提取适合注入给 Agent 的项目摘要
- 支持输出不同粒度的 context bundle
- 支持为特定任务生成上下文（例如 coding / review / debug / release）

适合人群：

- 使用代码 Agent、审查 Agent、自动化开发 Agent 的开发者

为什么适合插件化：

- 强依赖本地项目结构和工作流
- 不同团队对“上下文打包”的偏好差异很大
- 可以独立演进注入策略、摘要策略和过滤规则

实现难度：中

### 2. `agent-memory`

用途：为 Agent 提供本地长期记忆能力。

建议能力：

- 保存跨会话的重要偏好、约束和决策
- 记录项目约定、常见命令、已知坑点
- 支持按项目、按 workspace、按用户维度组织记忆
- 支持查询、过期、清理、导入导出

适合人群：

- 长期在同一代码库中与 Agent 协作的开发者和团队

为什么适合插件化：

- 记忆策略、存储后端和检索规则都可能长期变化
- 这类能力适合作为可选增强层，而不是 core 默认职责

实现难度：中

### 3. `agent-skills`

用途：管理适合 Agent 使用的 prompt / skill / workflow 资产。

建议能力：

- 管理本地与远程 skill 库
- 支持模板变量、参数化执行和版本管理
- 支持导入导出 prompt / skill 包
- 支持按任务类型快速查找可复用技能

适合人群：

- 经常复用 Agent 操作模板、团队内部沉淀最佳实践的用户

为什么适合插件化：

- 这是典型的独立资产层能力
- 适合快速试错、快速扩展，不适合放进 core

实现难度：中

### 4. `mcp-manager`

用途：统一管理 Agent 依赖的 MCP 服务与工具接入。

建议能力：

- 注册、启停、检查本地或远程 MCP server
- 查看可用工具列表与协议元信息
- 检查连接状态、认证配置、健康状态
- 支持导出标准化配置，供不同 Agent / CLI / IDE 复用

适合人群：

- 需要让 Agent 使用本地工具、第三方服务、私有平台能力的团队

为什么适合插件化：

- 协议接入、部署方式、环境依赖差异很大
- 工具注册和服务治理天然适合独立为插件

实现难度：中到高

### 5. `agent-eval`

用途：评测 Agent 在真实任务中的效果和稳定性。

建议能力：

- 批量运行评测集
- 对比不同 prompt、skill、工具链和模型配置
- 输出结构化报告（如 json / csv / html）
- 支持回归测试，验证修改后效果是否退化

适合人群：

- 在团队中正式引入 Agent，希望持续验证质量的用户

为什么适合插件化：

- 评测逻辑和指标体系会持续演进
- 不是所有用户都需要，但对进阶用户非常关键

实现难度：中到高

### 6. `session-replay`

用途：记录和回放 Agent 的执行轨迹。

建议能力：

- 导出会话轨迹、工具调用记录、关键输入输出
- 按 session 回放执行过程
- 对比同一任务在不同版本下的行为差异
- 用于调试失败案例和分析误判原因

适合人群：

- 需要调试 Agent 行为或做问题复盘的用户

为什么适合插件化：

- 执行轨迹格式、采样粒度和回放策略都需要独立演进

实现难度：中

### 7. `multi-agent-runner`

用途：编排多 Agent 协作的任务流。

建议能力：

- 定义角色分工（例如 planner / coder / reviewer / releaser）
- 定义任务拆分与串并行执行关系
- 输出执行日志和阶段结果
- 支持失败重试、人工接管和中间状态查看

适合人群：

- 任务复杂、希望把 Agent 纳入标准化工程流程的团队

为什么适合插件化：

- 工作流形态差异极大，天然适合插件化扩展

实现难度：高

## 建议优先级

如果只优先做 3 个，我建议顺序如下：

1. `agent-context`
2. `mcp-manager`
3. `agent-eval`

### 优先理由

#### `agent-context`

- 几乎所有 coding / review / debug 场景都会用到
- 最容易直接提升 Agent 的理解质量
- 能作为后续 memory、eval、workflow 等插件的基础能力

#### `mcp-manager`

- 可以直接增强 Agent 的可用工具面
- 有助于把外部服务、本地脚本、平台能力统一接到 Agent 侧
- 对后续生态扩展价值很高

#### `agent-eval`

- 能把“感觉好不好用”变成可持续比较的工程能力
- 对第一方插件方向验证和后续迭代都很重要

## 建议下一步

建议按以下顺序推进：

1. 先从 `agent-context`、`mcp-manager`、`agent-eval` 中选一个做 MVP
2. 为选中的方向补一份设计文档，至少明确：
   - `plugin_id`
   - 配置结构
   - CLI 子命令
   - 与 core 的职责边界
   - 本地状态存储方式
   - 测试范围
3. 若决定进入实现，再同步补充：
   - `packages/<plugin>/AGENTS.md`
   - README
   - registry 条目
   - 插件测试与生命周期测试

## 期望结果

通过这一批插件，`agent-kit` 可以更明确地走向 **Agent 工具平台**：不仅能安装和调用插件，还能围绕 Agent 的上下文准备、工具接入、评测优化、执行回放和协作编排形成更完整的第一方能力。
