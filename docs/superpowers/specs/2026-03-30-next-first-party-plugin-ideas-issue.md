# 下一批第一方插件方向建议

## Issue 标题

建议规划下一批第一方插件，扩展 `agent-kit` 的开发者工作流覆盖面

## 背景

当前 `agent-kit` 的定位已经比较清晰：它不是一个把所有功能都塞进 core 的大而全 CLI，而是一个以 `agent-kit` 为统一入口、以插件为主要扩展载体的可扩展平台。

现阶段第一方插件只有两个：

- `skills-link`：把本地 skills 目录按目录粒度链接到一个或多个 target 目录
- `opencode-env-switch`：通过 shell 环境变量切换 OpenCode profile

这两个插件已经验证了平台机制本身是成立的，但覆盖的用户群体还比较窄，更多是面向特定工作流。为了让 `agent-kit` 更像一个“开发者工作流插件平台”，建议开始规划下一批受众更广、复用价值更高的第一方插件。

## 目标

- 在不扩大 core 职责边界的前提下，扩展 `agent-kit` 的插件生态
- 优先选择适合插件化、而不适合并入 core 的能力
- 优先覆盖开发环境、配置同步、项目初始化、自动化工作流等更通用的场景

## 设计判断

结合当前仓库现状，我认为适合继续做成第一方插件的能力，应该同时满足这些特征：

- 有独立配置和生命周期
- 与某一类开发工作流强相关
- 需要单独演进，不适合放到 core
- 可以复用现有插件的交互式 CLI、多语言、配置文件、状态查询等模式

## 候选插件方向

### 1. `dev-env-manager`

用途：管理多套开发环境并快速切换。

建议能力：

- 管理多个开发 profile
- 切换 PATH、环境变量和相关配置
- 提供 `export` 输出，便于配合 shell 使用
- 提供 `wizard` 初始化和更新入口

适合人群：

- Python / Node / Go / 多语言开发者

为什么适合插件化：

- 与具体开发工作流强相关
- 与现有 `opencode-env-switch` 的能力模型接近
- 不是所有用户都需要，不应放进 core

实现难度：中

### 2. `dotfiles-sync`

用途：同步和管理本地 dotfiles / 开发环境配置。

建议能力：

- 管理 `.gitconfig`、`.zshrc`、`~/.config/...` 等配置文件
- 支持软链接、复制、冲突检测、备份恢复
- 支持模板变量替换
- 支持多 target 或多机器场景

适合人群：

- 需要统一本地开发环境的个人用户和团队

为什么适合插件化：

- 与 `skills-link` 的“源目录 -> 多目标”模型有天然相似性
- 策略差异很大，不适合做成 core 通用能力

实现难度：中

### 3. `template-gen`

用途：生成项目模板或插件脚手架。

建议能力：

- 生成 Python 项目模板
- 生成 `agent-kit` 插件模板
- 通过交互式 CLI 采集参数
- 支持后续扩展更多模板类型

适合人群：

- 需要快速启动项目或开发新插件的用户

为什么适合插件化：

- 模板体系天然适合独立演进
- 能反过来降低 `agent-kit` 生态扩展门槛

实现难度：低到中

### 4. `shell-integration`

用途：增强 shell 集成能力。

建议能力：

- 初始化 `zsh` / `bash` / `fish` 集成
- 安装自动补全脚本
- 检查当前 shell 集成状态
- 统一管理 shell wrapper 或 source 片段

适合人群：

- 所有 `agent-kit` CLI 用户

为什么适合插件化：

- shell 差异明显，适合独立演进
- 属于体验增强层，不是 core 的主职责

实现难度：中

### 5. `config-snapshot`

用途：为插件配置提供快照、对比和回滚能力。

建议能力：

- 备份插件配置
- 比较两个版本的差异
- 恢复到某个历史版本
- 标记稳定快照

适合人群：

- 经常调整配置、希望可回滚的用户

为什么适合插件化：

- 这是独立的配置治理能力，不应侵入 core

实现难度：低

### 6. `script-run`

用途：管理和运行常用自动化脚本。

建议能力：

- 注册脚本
- 参数化执行
- 保存执行历史
- 聚合日志输出

适合人群：

- 有大量本地自动化脚本的开发者和运维用户

为什么适合插件化：

- 工作流定制色彩强
- 明显属于插件层，而不是平台底层能力

实现难度：低到中

## 建议优先级

如果只优先做 3 个，我建议顺序如下：

1. `dev-env-manager`
2. `dotfiles-sync`
3. `template-gen`

### 优先理由

#### `dev-env-manager`

- 受众最广
- 与现有 `opencode-env-switch` 的设计思路最接近
- 最容易做出“立刻能感知价值”的插件

#### `dotfiles-sync`

- 与 `skills-link` 的模型高度相似
- 能覆盖更广泛的本地开发环境管理需求
- 技术实现路径相对清晰

#### `template-gen`

- 不只解决用户问题，也能帮助后续继续扩展插件生态
- 适合作为未来更多第一方 / 第三方插件的脚手架入口

## 建议下一步

建议按以下顺序推进：

1. 先从上述 3 个优先项里确定一个 MVP
2. 为选中的方向补一份设计文档，明确：
   - `plugin_id`
   - 配置结构
   - CLI 子命令
   - 与 core 的职责边界
   - 测试范围
3. 若决定进入实现，再补充：
   - `packages/<plugin>/AGENTS.md`
   - README
   - registry 条目
   - 生命周期测试与插件测试

## 期望结果

通过下一批第一方插件，`agent-kit` 可以从“已有两个偏特定场景的插件”，进一步成长为一个更完整的开发者工作流插件平台。
